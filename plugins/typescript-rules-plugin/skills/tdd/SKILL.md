# TDD（テスト駆動開発）ルール

## 基本理念

- **テストファースト**: 実装前に必ずテストを書く — テストが「仕様書」になり、実装の仕様からの逸脱を防ぐ
- **Red-Green-Refactor**: 失敗 → 成功 → 改善のサイクル — 各フェーズの目的を分離する。Redで仕様を定義し、Greenで最小実装に集中し、Refactorで設計を改善する
- **ベイビーステップ**: 小さな一歩ずつ進む — ステップが大きいとRedに戻れなくなる。小さいステップなら失敗時にすぐ直前の状態に戻れる
- **意図を表明**: テストでコードの意図を明確に示す

## TDDサイクル

1. **Red**: まず失敗するテストを書く
2. **Green**: テストが通る最小限のコードを書く
3. **Refactor**: 重複を排除し、コードを改善

## 三角測量

複数のテストケースを追加して仮実装を一般化に導く手法。

### 手順

1. 1つ目のテスト → 仮実装（固定値を返す）でGreenにする
2. 2つ目のテスト → 仮実装では通らない別の入力を与える
3. 仮実装を一般化して両方のテストを通す

### 仮実装から脱するタイミング

- 仮実装（固定値）で通るテストが1つだけ → まだ仮実装でよい
- 2つ目のテストが仮実装では通らない → 一般化する
- 一般化のアルゴリズムが自明（単純な四則演算など）→ 三角測量を省略して直接実装してもよい

## ベイビーステップの粒度

### 適切な粒度の目安

- 1ステップ = 1つのテストケース追加 + それを通す最小限の実装
- Greenまでの実装変更が10行を超えたらステップが大きすぎるサイン
- 迷ったら小さい方を選ぶ — 小さすぎて困ることはないが、大きすぎるとRedに戻れなくなる

### ステップを分割すべきサイン

- テストを通すために複数の関数を同時に変更する必要がある
- テストケース名に「AかつB」「AまたはB」が含まれる
- Greenにするまでに5分以上かかる

## 実行手順

1. 空の関数・クラスのスケルトンを作成
2. テストを書く
3. `pnpm run test` でテスト失敗を確認（Red）
4. テストが通る最小限のコードを実装（Green）
5. `pnpm run test` でテストパスを確認
6. 必要に応じてリファクタリング（Refactor）

## デイリーチェック

- [ ] 新機能は必ずテストから開始
- [ ] テストが失敗することを確認してから実装
- [ ] 仮実装から始めて段階的に一般化
- [ ] グリーンになったらリファクタを検討
- [ ] テストが文書として機能しているか確認
